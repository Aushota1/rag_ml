# LLM Configuration

## 📍 Где находится конфигурация LLM

LLM конфигурация **НЕ** находится в `.env` файле.

Все данные для подключения к LLM находятся в файле:
```
test_llm.py
```

## 🔧 Структура test_llm.py

```python
LLM_CONFIG = {
    "provider": "polza",
    "model": "gpt-5",
    "api_key": "pza_ваш_ключ",
    "base_url": "https://polza.ai/api/v1"
}
```

## 🧪 Запуск тестов

```bash
python test_llm.py
```

Это запустит 4 теста:
1. Connection - проверка подключения
2. JSON Response - проверка JSON ответов
3. With Context - проверка с контекстом
4. LLM Integration - проверка интеграции с llm_pipline.py

## ⚙️ Изменение конфигурации

Чтобы изменить настройки LLM, отредактируйте `LLM_CONFIG` в `test_llm.py`:

```python
LLM_CONFIG = {
    "provider": "openai",  # или "polza"
    "model": "gpt-4o-mini",  # или "gpt-5"
    "api_key": "ваш_новый_ключ",
    "base_url": "https://api.openai.com/v1"  # или другой URL
}
```

## 📁 Файлы

- `.env` - НЕ содержит LLM конфигурацию
- `.env.example` - НЕ содержит LLM конфигурацию
- `test_llm.py` - СОДЕРЖИТ LLM конфигурацию и тесты
- `llm_pipline.py` - модуль интеграции LLM

## 🔒 Безопасность

`test_llm.py` содержит API ключи и должен быть:
- Добавлен в `.gitignore`
- Не коммититься в репозиторий
- Использоваться только локально для тестов

## 💡 Использование в production

Для production использования создайте отдельный конфигурационный файл или используйте переменные окружения напрямую в коде.

## ✅ Преимущества такого подхода

1. LLM конфигурация изолирована от основной конфигурации
2. Легко тестировать разные настройки
3. Не нужно менять `.env` для тестов
4. Все тесты в одном месте
5. Безопаснее - не случайно не закоммитишь ключи в `.env`
