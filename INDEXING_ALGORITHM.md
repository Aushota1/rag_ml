# 🔍 Алгоритм построения индексов RAG системы

## Обзор

Процесс индексации преобразует сырые PDF документы в структурированное векторное пространство, оптимизированное для быстрого семантического поиска.

## 📥 Входные данные

### Что получаем на вход:
```
Папка с PDF документами:
├── law_document_1.pdf
├── law_document_2.pdf
├── case_document_1.pdf
├── case_document_2.pdf
└── ... (всего ~100-200 документов)
```

### Характеристики входных данных:
- **Формат:** PDF файлы
- **Тип контента:** Юридические документы (законы, судебные дела)
- **Язык:** Английский
- **Структура:** Иерархическая (Articles, Sections, параграфы)
- **Размер:** От нескольких страниц до сотен страниц

## 🔄 Этапы обработки

### ЭТАП 1: Парсинг документов (DocumentParser)

#### Вход:
- PDF файлы из папки `DOCUMENTS_PATH`

#### Процесс:

```python
for each PDF file:
    1. Открыть PDF с помощью pypdf
    2. Для каждой страницы:
        a. Извлечь текст методом extract_text()
        b. Если текст пустой/короткий → применить OCR (опционально)
        c. Сохранить текст страницы с номером
    3. Извлечь метаданные:
        - Название документа
        - Номер закона (Law No. X of YYYY)
        - Номер дела (CFI XXX/YYYY, CA XXX/YYYY)
        - Дата
        - Юрисдикция (DIFC)
```

#### Выход:
```python
{
    'doc_id': 'law_document_1',
    'pages': [
        {'page_num': 1, 'text': 'Article 1. Definitions...'},
        {'page_num': 2, 'text': 'Article 2. Scope...'},
        ...
    ],
    'metadata': {
        'title': 'Operating Law 2018',
        'law_number': 'Law No. 7 of 2018',
        'date': '2018',
        'jurisdiction': 'DIFC'
    }
}
```

#### Технические детали:
- **Библиотека:** pypdf для извлечения текста
- **OCR:** pytesseract + pdf2image (для сканированных документов)
- **Regex паттерны:** Для извлечения номеров законов и дел
- **Обработка ошибок:** Try-except для каждого файла

---

### ЭТАП 2: Структурный чанкинг (StructuralChunker)

#### Вход:
- Распарсенные документы из Этапа 1

#### Процесс:

```python
for each document:
    for each page:
        1. Разбить текст по структурным элементам:
            - Article X (статьи)
            - Section X (секции)
            - Нумерованные пункты (1., 2., 3.)
            - Буквенные пункты ((a), (b), (c))
        
        2. Для каждой секции:
            if длина > chunk_size (512 токенов):
                Разбить на подчанки с overlap (50 токенов)
            else:
                Оставить как есть
        
        3. Добавить метаданные к каждому чанку:
            - doc_id (идентификатор документа)
            - page (номер страницы)
            - hierarchy (иерархия: Article 5 → Section 2)
            - title, law_number, date и т.д.
```

#### Выход:
```python
[
    {
        'text': 'Article 1. Definitions\nIn this Law...',
        'metadata': {
            'doc_id': 'law_document_1',
            'page': 1,
            'hierarchy': ['Article 1'],
            'title': 'Operating Law 2018',
            'law_number': 'Law No. 7 of 2018'
        }
    },
    {
        'text': 'Article 2. Scope\nThis Law applies to...',
        'metadata': {
            'doc_id': 'law_document_1',
            'page': 2,
            'hierarchy': ['Article 2'],
            ...
        }
    },
    ...
]
```

#### Зачем нужен чанкинг:
1. **Ограничения моделей:** Эмбеддинг модели имеют лимит на длину текста
2. **Точность поиска:** Маленькие чанки = более точное совпадение
3. **Контекст:** Overlap сохраняет связность между чанками
4. **Структура:** Сохранение иерархии помогает в ранжировании

### ЭТАП 3: Создание эмбеддингов (HybridIndexer - Векторная часть)

#### Вход:
- Список чанков из Этапа 2 (~15000 чанков)

#### Процесс:

```python
1. Загрузить модель эмбеддингов:
    model = SentenceTransformer('all-MiniLM-L6-v2')
    # Размерность: 384

2. Извлечь тексты из чанков:
    texts = [chunk['text'] for chunk in chunks]

3. Создать эмбеддинги (батчами по 32):
    embeddings = model.encode(
        texts,
        batch_size=32,
        show_progress_bar=True
    )
    # Результат: numpy array [15000, 384]

4. Нормализовать векторы (для косинусного сходства):
    faiss.normalize_L2(embeddings)
    # Каждый вектор теперь имеет длину 1

5. Создать FAISS индекс:
    dimension = 384
    index = faiss.IndexFlatIP(dimension)  # Inner Product
    index.add(embeddings.astype('float32'))
```

#### Выход (векторный индекс):
```
FAISS Index:
- Тип: IndexFlatIP (Inner Product)
- Размерность: 384
- Количество векторов: 14929
- Размер в памяти: ~23 MB
- Формат на диске: vector.index
```

#### Что такое эмбеддинг:
- **Определение:** Плотное векторное представление текста
- **Размерность:** 384 числа (float32)
- **Свойство:** Похожие тексты → близкие векторы
- **Метрика:** Косинусное сходство (скалярное произведение после нормализации)

#### Пример эмбеддинга:
```python
text = "Article 1. Definitions"
embedding = [0.023, -0.145, 0.089, ..., 0.234]  # 384 числа
```

### ЭТАП 4: Создание BM25 индекса (HybridIndexer - Лексическая часть)

#### Вход:
- Те же тексты чанков

#### Процесс:

```python
1. Токенизировать каждый текст:
    tokenized_corpus = [
        text.lower().split() 
        for text in texts
    ]
    # Пример: "Article 1" → ["article", "1"]

2. Создать BM25 индекс:
    bm25_index = BM25Okapi(tokenized_corpus)
    # Вычисляет IDF (Inverse Document Frequency) для каждого слова

3. Сохранить статистику:
    - Частота слов в каждом документе (TF)
    - Обратная частота документов (IDF)
    - Средняя длина документа
```

#### Выход (BM25 индекс):
```
BM25 Index:
- Алгоритм: Okapi BM25
- Параметры: k1=1.5, b=0.75 (по умолчанию)
- Словарь: ~50000 уникальных токенов
- Размер на диске: bm25.pkl (~10 MB)
```

#### Что такое BM25:
- **Определение:** Алгоритм ранжирования на основе частоты слов
- **TF (Term Frequency):** Как часто слово встречается в документе
- **IDF (Inverse Document Frequency):** Насколько редкое слово в корпусе
- **Формула:** BM25(q,d) = Σ IDF(qi) × (f(qi,d) × (k1+1)) / (f(qi,d) + k1 × (1-b+b×|d|/avgdl))

#### Зачем нужен BM25:
1. **Точное совпадение:** Находит документы с конкретными словами
2. **Дополнение к векторам:** Векторы хороши для семантики, BM25 для точности
3. **Редкие термины:** Хорошо работает с юридическими терминами
4. **Быстрый:** Не требует нейросетей

---

### ЭТАП 5: Сохранение на диск

#### Процесс:
```python
1. Сохранить FAISS индекс:
    faiss.write_index(vector_index, 'index/vector.index')

2. Сохранить BM25 индекс:
    pickle.dump(bm25_index, 'index/bm25.pkl')

3. Сохранить чанки с метаданными:
    pickle.dump(chunks, 'index/chunks.pkl')

4. Сохранить тексты:
    pickle.dump(texts, 'index/texts.pkl')
```

#### Выход (файлы на диске):

```
index/
├── vector.index      # FAISS векторный индекс (~23 MB)
├── bm25.pkl         # BM25 индекс (~10 MB)
├── chunks.pkl       # Чанки с метаданными (~15 MB)

Общий размер: ~56 MB
```

## 🔍 Как используются индексы при поиске

### Гибридный поиск (Hybrid Search):

```python
def hybrid_search(query, top_k=40, alpha=0.5):
    # 1. Векторный поиск
    vector_results = vector_index.search(query, k=20)
    # Находит семантически похожие чанки
    
    # 2. BM25 поиск
    bm25_results = bm25_index.search(query, k=20)
    # Находит чанки с точным совпадением слов
    
    # 3. Нормализация оценок
    vector_scores_norm = normalize(vector_results)
    bm25_scores_norm = normalize(bm25_results)
    
    # 4. Комбинирование
    combined_scores = (
        alpha * vector_scores_norm + 
        (1 - alpha) * bm25_scores_norm
    )
    
    # 5. Сортировка и возврат top-k
    return sort_and_return(combined_scores, top_k)
```

### Параметры:
- **alpha = 0.5:** Равный вес векторному и BM25 поиску
- **top_k = 40:** Возвращаем 40 кандидатов для дальнейшего реранкинга

## 🔬 Технические детали

### FAISS IndexFlatIP

**Что это:**
- Flat = полный перебор (brute force)
- IP = Inner Product (скалярное произведение)

**Как работает:**

```python
1. Хранит все векторы в памяти
2. При поиске вычисляет скалярное произведение с каждым вектором
3. Возвращает top-k с наибольшими значениями

Сложность:
- Добавление: O(1)
- Поиск: O(n × d) где n=количество векторов, d=размерность
- Память: O(n × d × 4 bytes)
```

**Преимущества:**
- ✅ Точный (100% recall)
- ✅ Простой
- ✅ Детерминированный

**Недостатки:**
- ❌ Медленный для больших датасетов (>1M векторов)
- ❌ Требует много памяти

### BM25 Okapi

**Формула:**
```
BM25(q,d) = Σ IDF(qi) × (f(qi,d) × (k1+1)) / (f(qi,d) + k1 × (1-b+b×|d|/avgdl))

где:
- q = запрос
- d = документ
- qi = i-й термин в запросе
- f(qi,d) = частота термина qi в документе d
- |d| = длина документа d
- avgdl = средняя длина документа в коллекции
- k1 = параметр насыщения (обычно 1.2-2.0)
- b = параметр нормализации длины (обычно 0.75)
```

**Интуиция:**
1. **IDF:** Редкие слова важнее частых
2. **TF с насыщением:** Много повторений не всегда лучше
3. **Нормализация длины:** Длинные документы не должны доминировать

### Sentence Transformers

**Архитектура:**
```
Input Text
    ↓
Tokenization (WordPiece)
    ↓
BERT/RoBERTa Encoder
    ↓
Mean Pooling
    ↓
Dense Layer (опционально)
    ↓
L2 Normalization
    ↓
384-dim Embedding
```

**Модель all-MiniLM-L6-v2:**
- Базовая модель: MiniLM
- Слои: 6 (легкая версия)
- Параметры: ~22M
- Размерность: 384
- Скорость: ~2000 предложений/сек (CPU)

## 💡 Практические советы

### Когда перестраивать индекс:

✅ **Нужно перестроить:**
- Добавлены новые документы
- Изменены параметры чанкинга (chunk_size, overlap)
- Изменена модель эмбеддингов
- Обновлен контент существующих документов

❌ **Не нужно перестраивать:**
- Изменены параметры поиска (top_k, alpha)
- Изменен реранкер
- Изменен генератор ответов
- Изменены промпты для LLM

### Проверка качества индекса:

```python
# 1. Проверить количество чанков
print(f"Total chunks: {len(chunks)}")
# Ожидаемо: ~50-100 чанков на документ

# 2. Проверить размерность эмбеддингов
print(f"Embedding dimension: {embeddings.shape[1]}")
# Должно быть: 384

# 3. Проверить нормализацию
norms = np.linalg.norm(embeddings, axis=1)
print(f"Mean norm: {norms.mean()}")
# Должно быть: ~1.0

# 4. Тестовый поиск
results = hybrid_search("What is Article 1?", top_k=5)
print(f"Found {len(results)} results")
# Должно вернуть релевантные чанки
```

### Отладка проблем:

**Проблема:** Индекс не находит релевантные документы
```
Решения:
1. Проверить качество парсинга (есть ли текст?)
2. Проверить размер чанков (не слишком большие/маленькие?)
3. Попробовать другую модель эмбеддингов
4. Настроить параметр alpha (баланс vector/BM25)
```

**Проблема:** Индексация слишком медленная
```
Решения:
1. Увеличить batch_size (если есть память)
2. Использовать GPU
3. Уменьшить количество документов для теста
4. Кэшировать эмбеддинги
```

**Проблема:** Индекс занимает слишком много места
```
Решения:
1. Использовать квантизацию (IndexIVF)
2. Уменьшить размерность эмбеддингов (PCA)
3. Увеличить размер чанков (меньше чанков)
4. Сжать pickle файлы (gzip)
```

## 🎓 Заключение

Процесс индексации преобразует неструктурированные PDF документы в высокоэффективное векторное пространство, оптимизированное для семантического поиска. Гибридный подход (векторы + BM25) обеспечивает баланс между семантическим пониманием и точным совпадением терминов, что критично для юридических документов.

**Ключевые преимущества системы:**
- ✅ Быстрый поиск (миллисекунды)
- ✅ Семантическое понимание (векторы)
- ✅ Точное совпадение (BM25)
- ✅ Сохранение структуры (иерархия)
- ✅ Масштабируемость (до миллионов документов)

**Итоговая схема:**
```
PDF → Parser → Chunks → Embeddings → FAISS Index
                  ↓
                  └──→ BM25 Index

Результат: Гибридная система поиска
```
