# Анализ соответствия требованиям хакатона

## 📋 Требования хакатона

### 1. Детерминированная точность ✅

#### Требования:
- Числовой допуск (±1%)
- Точное совпадение для логических значений и имен
- Коэффициент сходства Жаккара для списков
- Точное соответствие датам по стандарту ISO 8601
- JSON null для отсутствующей информации

#### Статус реализации:

**✅ ВЫПОЛНЕНО:**

1. **Boolean (true/false/null)**
   - Файл: `llm_pipline.py`, строки 82-115
   - Реализация: LLM генерация с fallback эвристикой
   - Гарантия: ВСЕГДА возвращает true/false/null
   - Улучшенная логика: Подсчет позитивных/негативных слов, инверсия для отрицательных вопросов

2. **Number (числа с допуском ±1%)**
   - Файл: `llm_pipline.py`, строки 117-125
   - Реализация: LLM экстракция + regex fallback
   - Формат: int или float, null если не найдено
   - Валидация: `generate_submission.py` конвертирует в int/float

3. **Date (ISO 8601: YYYY-MM-DD)**
   - Файл: `llm_pipline.py`, строки 127-135
   - Реализация: LLM экстракция + regex fallback
   - Формат: строго "YYYY-MM-DD" или null
   - Пример: "2026-02-02"

4. **Name (одно имя)**
   - Файл: `llm_pipline.py`, строки 137-145
   - Реализация: LLM экстракция + regex fallback
   - Формат: строка или null

5. **Names (список имен - Jaccard similarity)**
   - Файл: `llm_pipline.py`, строки 147-155
   - Реализация: LLM экстракция + regex fallback
   - Формат: массив строк или пустой массив []
   - Дедупликация: `generate_submission.py` удаляет дубликаты

6. **Free_text (до 280 символов)**
   - Файл: `llm_pipline.py`, строки 157-162
   - Реализация: LLM генерация
   - Ограничение: 280 символов (обрезается в `generate_submission.py`)

7. **JSON null для отсутствия информации**
   - Файл: `generate_submission.py`, строки 60-95
   - Реализация: Все типы возвращают null при отсутствии данных
   - Проверка: has_info флаг в retriever.py

---

### 2. Качество свободного текста (оценка LLM) ✅

#### Требования:
- Правильность
- Полнота
- Заземление (grounding)
- Калибровка достоверности
- Ясность и краткость

#### Статус реализации:

**✅ ВЫПОЛНЕНО:**

1. **LLM Integration**
   - Файл: `llm_pipline.py`
   - Модель: `google/gemini-2.5-flash` (Polza AI)
   - Промпты: Специализированные для каждого типа ответа
   - Системный промпт: "You are a precise legal document assistant"

2. **Grounding (заземление)**
   - Файл: `llm_pipline.py`, метод `_build_context()`
   - Контекст: Собирается из top-5 reranked чанков
   - Метаданные: doc_id + page для каждого чанка
   - Полные страницы: `_get_full_page_text()` объединяет все чанки со страницы

3. **Retry механизм**
   - Файл: `llm_pipline.py`, строки 200-240
   - Попытки: До 2 retry при пустом ответе
   - Fallback: Эвристическая экстракция при ошибке LLM

4. **JSON парсинг**
   - Файл: `llm_pipline.py`, метод `_parse_llm_response()`
   - Валидация: Проверка структуры {"type": "...", "value": ...}
   - Fallback: Regex экстракция при ошибке парсинга

### 3. Оценка извлечения/закрепления ✅

#### Требования:
- Наказания за нерелевантные страницы (шум)
- Наказания за отсутствие необходимых доказательств (recall)
- Сбалансированная система

#### Статус реализации:

**✅ ВЫПОЛНЕНО:**

1. **Гибридный поиск (Precision + Recall)**
   - Файл: `indexer.py`, метод `hybrid_search()`
   - Векторный поиск: FAISS (семантическое сходство)
   - BM25 поиск: Лексическое совпадение
   - Комбинирование: α=0.5 (баланс)
   - Top-K: 1000 кандидатов на первом этапе

2. **Reranking (Precision)**
   - Файл: `reranker.py`
   - Модель: `cross-encoder/ms-marco-MiniLM-L-6-v2`
   - Попарная оценка: (query, document)
   - Top-K: 100 финальных результатов

3. **Threshold Validation (Noise reduction)**
   - Файл: `retriever.py`, строки 50-55
   - Порог: 0.05 (очень низкий для высокого recall)
   - Флаг: has_info = (max_score >= threshold)
   - Результат: Фильтрация нерелевантных результатов

4. **Relevance Classifier (опционально)**
   - Файл: `retriever.py`, строки 60-75
   - Модель: Tokenier relevance classifier
   - Порог: 0.3
   - Дополнительная фильтрация нерелевантных чанков

5. **Телеметрия retrieved_chunk_pages**
   - Файл: `pipeline.py`, строки 90-95
   - Формат: [{"doc_id": "...", "page_numbers": [1, 2, 3]}]
   - Группировка: По документам
   - Сортировка: Страницы по возрастанию

### 4. Задержка (модификатор TTFT) ⚠️

#### Требования:
- <1с → +5% бонус
- 1–2 с → +2%
- 2–3s → без модификатора
- >3 с → штраф до −15%

#### Статус реализации:

**⚠️ ЧАСТИЧНО ВЫПОЛНЕНО:**

1. **TTFT измерение**
   - Файл: `pipeline.py`, строка 75
   - Измерение: До генерации ответа (после retrieval)
   - Формат: Миллисекунды

2. **Текущая производительность (из submission.json):**
   - TTFT: 12,000-35,000 мс (12-35 секунд) ❌
   - Total time: 15,000-40,000 мс (15-40 секунд) ❌
   - **ПРОБЛЕМА**: Слишком медленно!

3. **Причины медленной работы:**
   - LLM вызовы: 10-30 секунд на запрос
   - Большой TOP_K_RETRIEVAL: 1000 чанков
   - Большой TOP_K_RERANK: 100 чанков
   - Нет кэширования
